# RF-8 and RF-9 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement `Analyze -> Users contribution` and `Analyze -> Most active days and hours`, including recursive folder discovery for non-repo directories and ASCII-table output across all discovered repositories.

**Architecture:** Keep repository discovery and aggregation in shared crates. Extend the existing single `git log --format=%H%x1f%an%x1f%ae%x1f%aI --numstat` pass so each repository produces both the current summary data and new per-user activity facts. Build RF-8 and RF-9 rows in `repolyze-core`, render ASCII tables in `repolyze-report`, and let CLI/TUI remain orchestration and presentation layers.

**Tech Stack:** Rust 2024 workspace, `clap`, `ratatui`, `git` subprocesses, `assert_cmd`, `tempfile`

---

## Confirmed Decisions

- Recursive folder discovery is in scope.
- If an input path is inside a Git repository, analyze that enclosing repository only.
- If an input path is a directory that is not inside a Git repository, scan downward for nested repositories.
- RF-8 and RF-9 use current behavior and analyze commits reachable from each repository's current `HEAD`, not `git log --all`.
- The first pass does not add persistent disk caching.

## Caching Notes

The data worth caching is not rendered ASCII tables. The best cache unit is per-repository contributor activity facts derived from the parsed commit stream:

- contributor totals: commits, lines added, lines deleted, files touched
- weekday commit counts per contributor
- hour commit counts per contributor
- distinct active dates per contributor
- distinct active dates per weekday per contributor
- distinct active date-hour buckets per contributor and per hour

If these facts are stored in `RepositoryAnalysis`-adjacent models, RF-8 and RF-9 can be recomputed without re-running `git log`. Persistent disk caching can be added later behind a repository-root plus `HEAD` fingerprint without changing the table renderers.

```rust
pub fn resolve_input(path: &Path) -> Result<Vec<RepositoryTarget>, RepolyzeError> {
    let canonical = path
        .canonicalize()
        .map_err(|_| RepolyzeError::PathNotFound(path.to_path_buf()))?;

    if let Ok(root) = find_git_root(&canonical) {
        return Ok(vec![RepositoryTarget { root }]);
    }

    if canonical.is_dir() {
        let roots = discover_git_roots(&canonical)?;
        if roots.is_empty() {
            return Err(RepolyzeError::NoRepositoriesFound(canonical));
        }
        return Ok(roots.into_iter().map(|root| RepositoryTarget { root }).collect());
    }

    Err(RepolyzeError::NotAGitRepository(path.to_path_buf()))
}
```

Then update `resolve_inputs` and `resolve_inputs_with_failures` to flatten the vectors returned per input.
